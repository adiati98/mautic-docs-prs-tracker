▶ New here? How to read this board

1. The four groups — whose turn is it?

Need you todayActions only you can take, most urgent first.
Bring it forwardNot urgent, but worth doing on your own schedule — brand-new PRs that still need a label or milestone, approvals that are ready to merge with nothing else going on, and anything that's gone quiet for a while (stale).
Waiting on others or for code PR to mergeYou've done your part — either the code PR hasn't merged yet, a reminder to the code PR author has been sent, or you've escalated and are waiting for a reply.
MonitoringThe author replied and you've already responded. Collapsed by default — if the conversation goes quiet for a week, it resurfaces so you can send another reminder.

2. Filters — narrowing what you see

Both filter bars live at the top of the page, above the groups.

RepoSwitches the whole board to show just one repo. The counts on each tab are totals across all four groups.
PriorityCritical, Serious, and Triage only apply to Need you today — picking one of these hides the other three groups entirely. Act and Stale work differently: they show up in every group instead of hiding the rest, since Bring it forward has its own actionable and stale rows too. Counts follow whichever repo tab is selected.
CriticalYou reminded the code author 14+ days ago and it's still quiet — escalate to the core team.
SeriousYou reminded the code author 7–14 days ago, or someone's waiting directly on your reply — send a follow-up.
ActSomething needs doing: check the author's response, remind the code author, review a standalone PR, do a final review on an approval that has something else attached, add a label/milestone to a new PR, or merge one that's cleanly approved and ready.
TriageReview a draft PR — still waiting the code PR to merge — or, for a standalone PR, waiting on its author before it escalates.
🕸 StaleNo activity on the PR for 30+ days.

3. Reading a row

Left edgeUrgency at a glance: red overdue → orange due soon → blue actionable → dark grey triage (needs your review) → green approved/ready to merge. A plain pale edge (no colour) means the row is in Waiting — nothing to do right now, it's on someone else — regardless of whether the code PR is a draft, open, or merged.
OpenA badge — a fact about the code PR: Draft / Open / Merged / Closed.
Review this docs PRA label — an action for you. Colour = the type of task (see below).

4. Colour key — same kind of task, same colour

Setup & triageAdd milestone (every new PR) · Add pending-pr-merge label (drafts) · Add needs-backport label (older branch)
Remind Follow up EscalateThe same colour, getting more intense the more urgent it gets: Remind → Follow up → Escalate.
Review / respondReview this docs PR (standalone, awaiting your review) · Check the author's response · Note since approval.
Finish & mergeFinal review, then merge · Remove pending-pr-merge label · Approved by X · Approved — ready to merge.
Backport firstMust be backported before it can merge.
Manual attentionNo code PR linked · someone's waiting on a reply · docs PR needs a rebase · code PR's milestone doesn't match the docs branch/milestone yet.
Optional / already doneReview draft PR · a reminder you already sent · X reviewed this (someone's looked, no approval yet).
Close / dismissClose this docs PR — its code PR was closed.
🕸 StaleNo activity on the PR for 30+ days.

5. Live threads & approvals

👀 Live threadsAn unanswered human comment on the docs PR. Orange if someone's waiting on you; also shown if someone outside the review team is waiting on the code author — checked separately, so a later unrelated reply to someone else can't hide it. Once the PR is approved, only what's been said since that approval counts — earlier comments don't. Otherwise, it's just there so you can keep an eye on it. Includes Promptless when it @-mentions a reviewer outside your team for feedback.
Approvals"Approved by X" is a label shown wherever there's an active approval — whether that's you, a teammate, or someone else (the one exception: a PR author can't approve their own PR, except for one admin account GitHub allows this for). "Final review, then merge" only appears once the code PR has merged. For a standalone PR with no code PR to wait on, your own approval combines both into one label: "Approved by X — ready to merge". Anything added after the approval gets its own "Note since approval" label.
Content-approved labelGitHub automatically dismisses an approval the moment new commits land — even if the push only addresses unrelated feedback. The tracker keeps trusting the approval anyway ("Approved by X" and everything it unlocks stay put), but flags it with an "Add content-approved label" reminder so you can go confirm on GitHub that the new commits didn't actually change what was approved. Add the content-approved label there and the reminder clears.

6. Follow-up & escalation timeline

Only a comment that @-mentions the code author starts this timeline — yours or a teammate's, not just a regular reply — and it stops the moment the author replies.

Day 0Code PR merges, someone @-mentions the code author.
Day 7Row asks for a follow-up.
Day 14Row asks you to escalate to the core team.
Any dayAuthor replies — the row asks you to check their response.

7. Good to know

Review vs. the timelineThe remind/follow-up/escalate timeline no longer waits on you having formally reviewed the docs PR — it only needs the code PR to have merged. If review's still outstanding, a separate "Review this docs PR — code PR merged" label shows up alongside whatever the timeline shows.
Review vs. triageA brand-new PR still missing its label/milestone also gets a "Review this docs PR" label right away, alongside the setup chips — reading the content doesn't have to wait on triage being finished.
Labelspending-pr-merge — removed once the code PR merges. needs-backport — added when a PR targets an older branch than the latest. needs-rebase — just shown as-is, it doesn't affect timing. content-approved — clears the "Add content-approved label" reminder once you've confirmed a GitHub-dismissed approval still stands; see "Content-approved label" above.

8. Your personal checklist

Every row in Need you today, Bring it forward, and Waiting has a checkbox — Monitoring doesn't, since it's already "nothing to do."

Tick it off once you've actually handled a row. It's saved to your own browser only — nobody else sees it, and it won't change anything on GitHub or on this page's data. The row dims and its title gets struck through; clearing your browser data resets everything.
2/5 checkedEach group's header shows its own progress, always out of the whole group — not just whatever's currently filtered into view. Grey while untouched, blue while in progress, green with a ✓ once every row in that group is checked off.
Hide checked rowsA switch next to the Repo/Priority filters (Checklist row) that collapses checked rows out of view instead of just dimming them. Your checked state is exactly the same either way — this only changes what's shown.
